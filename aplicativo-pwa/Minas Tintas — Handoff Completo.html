Regras Globais de Bônus

O bônus é o coração do programa de fidelidade dos pintores. Vale para todo o sistema.

Fórmula do bônus

O bônus é 1% do valor bruto do pedido em R$, convertido à proporção R$ 1 = 10 pts — o que equivale a 10% do total em pontos.

Exemplo: pedido de R$ 3.480 → 348 pts. O app exibe o atalho visual R$ [total] × 10% = [pts].

Cálculo de referência: bonus_pts = Math.round(total × 0.10)

O desconto NUNCA afeta o bônus

O bônus é sempre calculado sobre o subtotal bruto (antes do desconto). O desconto ao cliente não reduz os pontos do pintor. O total gravado no pedido é o valor bruto; o desconto fica em campo separado.

Bônus × status do pedido

StatusComportamento do bônus
PendenteBônus não creditado — apenas previsto ("a creditar")
AprovadoBônus creditado automaticamente no saldo do pintor, no momento da aprovação
RecusadoNenhum bônus gerado — os pontos nunca chegaram a ser creditados
EstornadoBônus que foi creditado é revertido (debitado) do saldo do pintor
Multiplicador da lojinha (≠ bônus)

Não confundir: o bônus de pedido (10%) é fixo. A lojinha tem um multiplicador_global separado (padrão 10) que converte o custo de cada item em pontos de resgate. Ver seção 7.

Contratos Globais

Regras que valem para toda a integração entre os dois apps e sistemas externos.

Autenticação do Admin

  • Login via e-mail + senha. Sem OAuth.
  • Usar Supabase Auth (supabase.auth.signInWithPassword).
  • Sessão de 30 dias — "lembrar-me" ativo por padrão. Após 30 dias, o admin re-autentica com e-mail e senha.
  • O Admin opera via service role key ou política com is_admin = true — acesso irrestrito (sem RLS de isolamento por pintor).

Integração ADM ↔ Pintor

Os dois apps compartilham o mesmo banco Supabase. Não existe fila de eventos separada. O ADM atualiza tabelas compartilhadas; o Pintor lê essas mesmas tabelas.

Aprovar pedidoInserir lançamento em pontos_movimentos (crédito). Inserir linha em notificacoes para o pintor (tipo: 'pedido_aprovado').
Recusar pedidoInserir linha em notificacoes para o pintor (tipo: 'pedido_recusado') — se decidida como necessária.
Estornar pedidoInserir lançamento em pontos_movimentos (débito estorno).
Confirmar entrega de resgateInserir linha em notificacoes para o pintor (tipo: 'resgate_confirmado').
Responsabilidade das notificações

A notificação é sempre criada pelo backend no momento da ação do Admin. O app Pintor nunca cria notificações — só as consome via GET /pintores/me/notificacoes.

Entidades Principais

Entidades compartilhadas entre Admin e Pintor no mesmo banco Supabase.

  • Admin — usuário do painel, com nome, e-mail, senha e foto.
  • Pintor — parceiro cadastrado pelo Admin, com nome, CPF, telefone, cidade e status.
  • Cliente — pessoa física ou empresa vinculada a um ou mais pintores.
  • Pedido — orçamento criado pelo pintor ou pelo Admin, com itens, pagamento e status.
  • ItemPedido — produto, quantidade, preço unitário e subtotal dentro do pedido.
  • ProdutoCatalogo — produto do Hiper Loja, usado na composição de orçamentos.
  • ProdutoLoja — item cadastrado na lojinha para resgate com pontos.
  • Resgate — solicitação de troca de pontos por produto da lojinha.
  • PontosMovimento — lançamento de crédito/débito/estorno de pontos do pintor.
  • Notificacao — evento gerado pelo backend para consumo do app Pintor.
  • Configuracao — parâmetros globais do sistema (multiplicador, percentual de bônus).

Relacionamentos

  • Pedido pertence a um pintor e a um cliente.
  • Cliente pode ser vinculado a mais de um pintor.
  • Resgate pertence a um pintor e a um produto da lojinha.
  • Todo crédito, débito ou estorno de pontos deve gerar PontosMovimento.
  • Notificações pertencem ao pintor — criadas sempre pelo backend em resposta a ações do Admin.

Decisões Supabase

Decisões técnicas de banco e infraestrutura. Não reabrir sem motivo.

Autenticação

  • Admin usa Supabase Auth com e-mail e senha (supabase.auth.signInWithPassword).
  • Pintor usa número de telefone + senha. O telefone é normalizado (somente dígitos) e usado como identificador em auth.users. Ver handoff do Pintor para detalhes.
  • A tabela pintores referencia auth.users via pintores.auth_user_id.
  • O Admin cadastra o pintor e define a senha inicial — o pintor altera por suporte (sem fluxo público de recuperação nesta fase).

Storage de imagens

  • Usar Supabase Storage para imagens dos itens da lojinha. Fluxo: Admin cadastra o produto e sobe a imagem → imagem é convertida para WebP e otimizada no momento do upload → arquivo armazenado no bucket → banco guarda o caminho/URL resultante.
  • Nunca guardar o binário no banco — apenas o caminho do arquivo no Storage.
  • Supabase Storage permite transformações (resize/quality) para servir thumbnails sem duplicar arquivos.

Campos úteis em produtos_loja: imagem_path · imagem_url · imagem_alt · imagem_bucket

Tela 1

Login

/login Admin · Desktop

Acesso exclusivo do perfil Admin via e-mail e senha. Sem OAuth.

Comportamento

O admin entra no painel digitando e-mail e senha. Não existe cadastro de admin pela interface nem login social — as contas são criadas manualmente no backend. Ao enviar, o backend valida as credenciais: se corretas, gera um token de sessão e devolve os dados do admin (nome e foto) para montar o cabeçalho do painel.

O token deve ser persistido e enviado em todas as requisições seguintes; quando faltar ou expirar, o usuário é mandado de volta para esta tela. Se as credenciais estiverem erradas, retorna 401 — sem dizer se foi o e-mail ou a senha que errou.

Início — Busca de Clientes

/dashboard Admin · Desktop

Tela de entrada. Verifica se um cliente já tem cadastro antes de criar um pedido — e é o hub de gestão de clientes (cadastrar e editar).

Comportamento

É a porta de entrada do admin. Antes de criar um pedido, ele verifica aqui se o cliente já existe: ao digitar nome ou telefone, recebe resultados em tempo real, cada um indicando se aquele cliente já tem um pintor vinculado.

Se o cliente não existe, o próprio campo de busca vira o atalho para cadastrá-lo — com o termo digitado já preenchido no formulário. Clicar num resultado abre o mesmo formulário em modo de edição. Cadastro e edição compartilham o componente: endereço auto-preenchido por CEP (ViaCEP) e deduplicação obrigatória por telefone.

Campo de busca

  • Busca simultânea por nome (qualquer quantidade de caracteres) e por telefone (a partir de 4 dígitos), em tempo real.
  • Retorna até 8 resultados.
  • Os chips Nome · Telefone · Empresa são decorativos (indicam critérios aceitos), não filtros interativos.

Tela 4

Pedido — Detalhe

/pedidos/[id] Admin · Desktop

Itens do orçamento, total, pagamento, bônus calculado, linha do tempo e ações conforme o status.

Comportamento

Onde o admin decide o destino de um pedido. Exibe os itens, o total, o bônus calculado e uma linha do tempo cumulativa de tudo que aconteceu. As ações disponíveis dependem do status: um pedido pendente pode ser aprovado ou recusado; um aprovado pode ser estornado, sempre com motivo obrigatório.

Lojinha

/lojinha Admin · Desktop

Duas abas: Produtos (catálogo de itens para resgate) e Resgates (solicitações dos pintores, pendentes ou finalizadas).

Comportamento

Onde o admin administra o catálogo de recompensas e processa os resgates. Na aba Produtos, cada item tem um custo que é convertido em pontos por um multiplicador global ajustável somado a um modificador individual — modificador negativo transforma o item em promoção automaticamente.

Na aba Resgates, o admin vê as solicitações dos pintores e confirma a entrega ou recusa. Pontos e estoque são debitados no momento da solicitação, não na entrega: se o admin recusa, ambos voltam ao pintor; se confirma a entrega, nada muda nos pontos (já foram debitados).

Banco de Dados

Decisões fechadas

O que foi decidido e molda o schema. Não reabrir sem motivo.

  • Auth: Supabase Auth com e-mail + senha para os dois perfis. Telefone é dado cadastral — não credencial.
  • Idioma: tabelas e colunas em português.
  • Pintor → auth: pintores.auth_user_id referencia auth.users(id).
  • Imagens: Supabase Storage (bucket). Banco guarda o caminho/URL — nunca o binário.

  • Cliente editado pelo pintor: salva direto, sem moderação.
  • Resgate: ao solicitar → debita pontos + estoque na hora; fica pendente até admin confirmar; cancelar → devolve tudo em transação.
  • Auditoria: logs_auditoria registra ações sensíveis (pontos, estoque, pedidos, resgates, status, permissões).
